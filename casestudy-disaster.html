재난 대응 시뮬레이터

"재난 유형별 대피소 추천 — 지도·점수·재난문자 시뮬레이션"

왜 만들었나

재난 문자는 받는데 어디로 가야 하는지 모른다. "폭우 주의보 발령"이라는 문자 한 줄이 전부다. 가장 가까운 대피소가 어디인지, 지금 운영 중인지, 내 상황에 맞는 곳인지 — 실제로 필요한 정보는 빠져 있다.

폭우와 지진은 최적 대피소가 다르다. 폭우엔 고지대·침수 대비 시설이 맞고, 화학사고엔 밀폐 가능한 고층 건물이 맞다. 재난 유형을 반영한 점수 기반 추천이 필요했다.

시뮬레이터라는 형식을 선택한 이유는 포트폴리오이기 때문이다. 실제 재난 API는 없으니 모의 데이터와 타이머로 긴박감을 연출하되, 알고리즘과 UX는 실제 서비스처럼 설계했다.

재난 유형 4종
대피소 20개
점수 요소 5개
코드 ~1,540줄

고민과 결정

설계 과정에서 맞닥뜨린 네 가지 선택지

결정 1 — 지도 라이브러리

고민: 지도를 어떤 라이브러리로 그릴까

A. Leaflet.js (오픈스트리트맵)  |  B. 카카오맵 / 네이버맵 API

→ 선택: A (Leaflet.js)

이유: 카카오·네이버 맵은 API 키 발급과 도메인 등록이 필요하다. 포트폴리오는 로컬·Vercel 양쪽에서 동작해야 하고 API 키를 코드에 노출할 수 없다. Leaflet은 CDN 하나로 무료로 쓸 수 있고, 오픈스트리트맵 타일에 CSS 필터(brightness(0.45) saturate(0.6) hue-rotate(200deg))를 입혀 다크 테마 지도를 만들었다.

결정 2 — 다크 전용 테마

고민: 다른 툴처럼 라이트/다크 토글을 달아야 하나

A. 다크 전용 (토글 없음)  |  B. 다른 툴처럼 라이트/다크 토글

→ 선택: A (다크 전용)

이유: 재난 상황 시뮬레이션은 긴박감이 핵심 UX다. 어두운 배경에 빨간 경보·파란 대피소·보라 추천 마커가 시각적으로 뚜렷하게 분리된다. 밝은 테마에서는 위험 구역 색상이 구분이 어렵고 긴박감도 사라진다. 지도 앱 대부분이 야간 모드를 기본으로 쓰는 이유와 같다.

결정 3 — 대피소 점수 알고리즘

고민: "가장 가까운 대피소"가 항상 최선인가

A. 단순 거리 기반 추천  |  B. 재난 유형·수용 여유·운영 상태 복합 점수

→ 선택: B (복합 점수)

이유: 가장 가까워도 이미 꽉 찼거나, 재난 유형에 부적합하거나, 운영이 중단됐을 수 있다. calcScore() 함수로 거리(40점)·수용 여유(25점)·재난 유형 적합도(20점)·운영 상태(10점)·고층 보너스(5점)를 합산한다. 화학사고·한파는 3층 이상 시설에 보너스 5점을 더해 재난 특성을 반영했다.

결정 4 — 재난문자 시뮬레이션

고민: 실제 재난문자 API가 없는데 어떻게 긴박감을 연출할까

A. 시드 데이터 + setTimeout 시퀀스  |  B. 정적 텍스트만 표시

→ 선택: A (타이머 기반 시퀀스)

이유: 재난 유형을 선택하면 속보 티커가 순서대로 흘러가고, 일정 시간 후 재난문자 팝업이 뜬다. 정적 텍스트는 "시뮬레이터"가 아니라 "정보 페이지"가 된다. 4가지 재난 유형마다 실제 같은 문구와 타임스탬프를 사전 작성해 시드로 심었다. 보는 사람이 실제 상황처럼 느끼도록 하는 것이 목표였다.

UX 결정

위험 구역 3단계 시각화

위험(빨강)·경계(주황)·주의(노랑) 세 단계로 원형 오버레이를 그린다. 반투명 원이 겹칠 수 있어 fillOpacity: 0.12로 낮게 설정하고 테두리만 진하게 강조했다. 사용자가 자신의 위치와 위험 구역의 관계를 직관적으로 파악하게 하는 것이 목표다.

슬라이드 패널 — transform 트랜지션

우측 결과 패널은 transform: translateX(100%) → translateX(0) 트랜지션으로 슬라이드 인한다. display: none 토글 대신 transform을 쓰는 이유는 CSS transition이 적용되기 때문이다. 지도와 패널이 공존하면서 지도를 완전히 가리지 않도록 패널 너비를 340px로 제한했다.

지도 다크 테마 — CSS 필터

Leaflet 타일에 직접 filter: brightness(0.45) saturate(0.6) hue-rotate(200deg)를 적용한다. 별도 다크 타일 서버 없이 CSS 한 줄로 네이비 계열 다크 맵을 만드는 방법이다. 마커와 오버레이는 필터 영향을 받지 않아 색상을 유지한다.

회고

지도 라이브러리 위에 커스텀 레이어를 쌓는 방식은 생각보다 까다로웠다. Leaflet의 마커·서클·폴리라인이 각자 다른 레이어에 올라가기 때문에 재난 유형이 바뀔 때마다 이전 레이어를 깔끔하게 제거해야 했다. 레이어 그룹을 변수로 관리하면서 clearLayers()를 명시적으로 호출하는 패턴이 해법이었다.

다크 전용 테마는 좋은 선택이었다. 토글을 구현하지 않아도 된다는 개발 비용 절감도 있었지만, 무엇보다 지도 위 색상 레이어의 가독성이 라이트 테마에서는 확보되지 않았다. 기능의 UX가 디자인 선택을 결정한 케이스였다.
