백테스트 비교기

"최대 3종목, 같은 기간 수익률을 나란히"

왜 만들었나

ETF와 주식을 고를 때 항상 같은 질문이 생긴다. "TQQQ와 UPRO, 같은 기간에 넣었으면 어느 쪽이 더 벌었을까?" 무료 백테스트 툴은 있지만 한국어 인터페이스가 없고, 영문 툴은 사용법이 복잡하거나 원하는 기간 설정이 불편했다.

직접 만들기로 했다. Yahoo Finance 실시간 데이터 + CORS 프록시 조합으로 빌드 도구 없이 구현 가능했다. 핵심 니즈는 단순했다: 종목 티커를 입력하고, 기간을 고르고, 수익률 추이를 같은 차트에서 보는 것.

개발 기간 ~2일
최대 종목 3개
비교 지표 5개
코드 ~1,200줄

제약과 출발점

포트폴리오의 모든 툴은 빌드 없는 단일 HTML이다. 백테스트는 금융 데이터가 필요한데, 무료 실시간 API가 제한적이다. Yahoo Finance v8 API가 무료지만 CORS 제약이 있어 브라우저에서 직접 호출이 불가능하다.

이미 공포탐욕지수에서 Vercel 서버리스 프록시(api/proxy.js)를 만들어두었다. 같은 프록시를 재사용해 Yahoo Finance 데이터를 끌어오면 CORS 문제가 해결되고, 추가 인프라 없이 백테스트 기능을 구현할 수 있었다.

고민과 결정

설계 과정에서 맞닥뜨린 네 가지 선택지

결정 1 — 종목 수 설계

고민: 종목을 몇 개까지 비교할 수 있게 할 것인가

A. 2종목 고정  |  B. 최대 3종목 동적 추가  |  C. 무제한

→ 선택: B (최대 3종목 동적)

이유: 2종목은 "A vs B" 비교만 가능하다. 실제로 "TQQQ vs UPRO vs QLD"처럼 3종목 비교가 필요한 순간이 많다. 무제한은 차트가 복잡해지고 API 호출이 많아진다. 3이 실사용 니즈와 차트 가독성의 균형점이다. 종목 C는 동적으로 추가/제거하며 UI 변화에 맞춰 차트와 통계 테이블 열이 실시간으로 조정된다.

결정 2 — 데이터 소스

고민: 과거 주가 데이터를 어디서 가져올 것인가

A. Yahoo Finance v8 API (+ Vercel 프록시)  |  B. 목 데이터 하드코딩  |  C. Alpha Vantage 무료 플랜

→ 선택: A (Yahoo Finance + Vercel 프록시)

이유: 목 데이터는 포트폴리오 도구로서 가치가 없다. Alpha Vantage 무료 플랜은 분당 5회 호출 제한이 있어 3종목 동시 조회가 불가능하다. Yahoo Finance는 제한이 없고, 기존 공포탐욕지수에서 이미 검증된 조합(CORS_PROXY + encodeURIComponent)을 그대로 재사용할 수 있어 추가 개발 비용이 없었다.

결정 3 — 비교 지표 선택

고민: 어떤 지표로 종목을 비교할 것인가

A. 총수익률 + CAGR + MDD + Sharpe + 변동성 (5개)  |  B. 총수익률만  |  C. 사용자가 선택

→ 선택: A (5개 고정)

이유: 수익률만 보면 위험을 놓친다. MDD(최대 낙폭)는 "이 종목을 계속 들고 버틸 수 있나"를 판단하는 가장 직관적인 지표다. CAGR은 기간을 통일해 비교할 수 있고, Sharpe는 위험 대비 수익을 단일 숫자로 표현한다. 5개 지표가 있으면 고수익·저위험 종목을 한눈에 가릴 수 있다. 사용자 선택은 UI가 복잡해진다.

결정 4 — 날짜 선택 UX

고민: 시작일·종료일 입력을 어떻게 구현할 것인가

A. flatpickr CDN (캘린더 피커)  |  B. HTML native <input type="date">  |  C. 프리셋 버튼만 (1Y / 3Y / 5Y)

→ 선택: A (flatpickr) + C (프리셋 버튼)

이유: Native date input은 브라우저별 UI가 달라 예측이 불가능하다. 특히 Safari에서 스타일 커스터마이징이 어렵다. flatpickr는 CDN 한 줄로 일관된 캘린더 UX를 제공한다. 프리셋 버튼(1Y / 3Y / 5Y / 최대)을 함께 제공해 대부분의 경우 날짜를 직접 입력하지 않아도 되도록 했다.

화면 설계

동적 3종목 구조와 지표 계산 로직

레이아웃 구조

backtest.html ├── topnav ├── 종목 입력 패널 │ ├── 종목 A [티커 입력] [색상: 파랑] │ ├── 종목 B [티커 입력] [색상: 빨강] │ └── 종목 C [티커 입력] [색상: 초록] ← 동적 추가/제거 │ + [+ 종목 추가] 버튼 ├── 기간 설정 (flatpickr 시작일·종료일 + 프리셋 버튼) ├── [비교하기] 버튼 ├── 수익률 라인 차트 (Chart.js, 기준일 100) ├── 통계 카드 행 (CAGR / MDD / 총수익률 / Sharpe / 변동성) └── 비교 테이블 (종목 × 지표, 1위 강조)

비교 지표 계산

지표계산 방법의미
총수익률 (최종가 - 시작가) / 시작가 × 100 기간 내 절대 수익
CAGR (최종가 / 시작가)^(1/연수) - 1 연환산 복리 수익률
MDD 고점 대비 최대 낙폭 (rolling max) 최악의 손실 시나리오
Sharpe (CAGR - 무위험수익률) / 연간 표준편차 위험 대비 수익 효율
변동성 일간 수익률의 표준편차 × √252 가격 등락의 심각성

동적 3종목 설계

종목 C 동적 추가/제거

종목 A·B는 고정이고 종목 C만 동적이다. "＋ 종목 추가" 클릭 시 입력 필드가 나타나고, 비교 차트에 세 번째 라인이 추가되며, 통계 카드와 비교 테이블에 열이 하나 더 생긴다. 제거 시 역순으로 모두 정리된다. 상태를 배열로 관리해 A·B·C 여부와 관계없이 동일한 렌더 로직이 적용된다.

비교 테이블 — 1위 vs 꼴찌 diff

각 지표(CAGR, MDD 등)별로 1위 종목을 자동 강조한다. diff 열은 1위와 꼴찌의 차이를 표시해 "얼마나 차이 나는가"를 직관적으로 보여준다. MDD는 낮을수록 좋으므로 최솟값이 1위다.

차트 기준일 정규화

세 종목의 절대 가격이 달라 단순 가격 비교는 의미가 없다. 시작일 가격을 100으로 정규화해 같은 축에 표시한다. 이렇게 하면 "같은 날 100만 원을 넣었다면" 시나리오를 직접 비교할 수 있다.

회고

동적 3종목 구조는 처음에 종목 A·B·C를 각각 독립 변수로 다루다가 버그가 많았다. 종목 데이터를 배열 tickers[0..2]로 통일하고 렌더 로직이 배열 길이에만 의존하도록 바꾼 뒤 코드가 크게 단순해졌다.

비교 테이블에서 MDD의 1위 기준(낮을수록 좋음)이 다른 지표(높을수록 좋음)와 반대라는 점을 나중에 발견했다. 지표마다 "방향성(direction)"을 명시적으로 선언하는 방식으로 수정해 비슷한 버그가 다시 생기지 않도록 했다.
