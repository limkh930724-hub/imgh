자산 계산기

"복리·목표·매매일지를 한 페이지로 통합"

왜 만들었나

투자를 시작하면 자연스럽게 세 가지 질문이 생긴다. "지금 넣으면 몇 년 뒤에 얼마가 될까?" "목표 금액을 만들려면 얼마씩 넣어야 하나?" "오늘 산 주식의 매수 단가와 수익률은?" 이 질문들에 답하는 도구를 하나씩 만들다 보니 세 개의 독립 페이지가 생겼다.

문제는 사용자(나 자신)가 세 도구를 오가는 동선이 불편했다는 것. 복리 계산을 마치고 바로 목표 시뮬레이션으로 넘어가고 싶은데, 매번 새 탭을 열어야 했다. 그래서 세 계산기를 하나의 허브로 통합하기로 했다.

개발 기간 ~2일
통합 탭 3개
구현 기능 8개
코드 ~664줄

기존 구조의 문제

처음엔 compound.html, goal.html, journal.html 세 파일이 따로 존재했다. 각각은 작동했지만 세 도구 사이에 연결고리가 없었다. 복리 계산 결과를 목표 시뮬레이션에 연결하려면 숫자를 손으로 옮겨야 했다. 통합 허브를 만들고 세 파일을 ?tab= 파라미터를 가진 리다이렉트 스텁으로 전환했다.

고민과 결정

설계 과정에서 맞닥뜨린 네 가지 선택지

결정 1 — 통합 방식

고민: 세 계산기를 어떻게 하나의 페이지에 담을 것인가

A. 탭 통합 허브 (asset.html + ?tab= 쿼리 파라미터)  |  B. 스크롤 단일 페이지  |  C. 멀티페이지 유지

→ 선택: A (탭 통합 허브)

이유: 스크롤 단일 페이지는 한 계산기를 쓸 때 다른 두 개가 시야에 들어와 집중을 방해한다. 멀티페이지 유지는 기존 문제를 해결하지 못한다. 탭은 각 도구를 독립적으로 사용하면서도 전환이 즉각적이다. URL 쿼리 파라미터로 탭을 식별하면 북마크와 직접 링크도 지원된다.

결정 2 — 탭 상태 관리

고민: 탭 전환 상태를 어디에 저장할 것인가

A. URL 쿼리 파라미터 (?tab=compound)  |  B. localStorage  |  C. JS 변수

→ 선택: A (URL 쿼리 파라미터)

이유: localStorage는 세션 간 상태를 유지하지만 링크 공유가 불가능하다. JS 변수는 새로고침 시 초기화된다. URL 파라미터는 "자산계산기 복리 탭 바로 열기" 링크를 만들 수 있고, 기존 compound.html·goal.html·journal.html을 리다이렉트 스텁으로 전환할 때도 자연스럽다.

결정 3 — 차트 라이브러리

고민: 복리 성장과 목표 시뮬레이션 차트를 어떻게 그릴 것인가

A. Chart.js CDN  |  B. Vanilla Canvas  |  C. SVG

→ 선택: A (Chart.js CDN)

이유: 복리 계산은 연도별 데이터 포인트가 많고, 호버 툴팁·범례·반응형 리사이즈가 필요하다. Vanilla Canvas로 구현하면 이 기능들을 모두 직접 만들어야 한다. Chart.js는 CDN 한 줄로 모든 기능을 가져오고, 포트폴리오 제약(단일 HTML, no build)에도 부합한다.

결정 4 — 매매일지 데이터 구조

고민: 날짜별 매수·매도 기록을 어떻게 저장할 것인가

A. 날짜 인덱스 객체 { 'YYYY-MM-DD': [...entries] }  |  B. 단순 배열  |  C. 종목별 분리 저장

→ 선택: A (날짜 인덱스 객체)

이유: "오늘 산 것 보기"와 "특정 날짜 기록 조회"가 핵심 사용 패턴이다. 배열이면 날짜 조회마다 전체를 순회해야 한다. 날짜를 키로 쓰는 객체는 O(1) 조회가 가능하고, localStorage JSON 직렬화도 자연스럽다. 종목별 분리는 "오늘 무엇을 샀는지"라는 시간 중심 질문에 맞지 않는다.

화면 설계

3탭 구조와 URL 기반 상태 관리

페이지 구조

asset.html ├── topnav (다크 고정 헤더, 각 탭 링크) ├── tab-bar (?tab= 파라미터로 활성 탭 결정) │ ├── [복리 계산] ?tab=compound │ ├── [목표 자산] ?tab=goal │ └── [매매일지] ?tab=journal └── tab-content (활성 탭만 display:block) compound.html → location.replace('/asset.html?tab=compound') goal.html → location.replace('/asset.html?tab=goal') journal.html → location.replace('/asset.html?tab=journal')

탭별 기능

탭입력출력영속
복리 계산 원금 / 연이율 / 기간(년) 연도별 복리 성장 라인 차트 + 최종 금액 없음 (입력값만)
목표 자산 목표금액 / 기간 / 현재 원금 / 연이율 필요 월 납입액 역산 + 누적 그래프 없음
매매일지 날짜 / 종목명 / 매수가 / 수량 / 현재가 수익률 · P&L · 평균단가 localStorage('fg-journal')

UX 결정

탭 초기화 — initAssetTabs() 단일 진입점

URL 파라미터 파싱, 탭 버튼 활성화, 콘텐츠 표시를 initAssetTabs() 하나가 담당한다. 파라미터가 없으면 compound를 기본값으로 한다. 탭 전환 시 URL을 history.replaceState로 업데이트해 뒤로 가기 히스토리를 오염시키지 않는다.

리다이렉트 스텁 — 기존 링크 보존

compound.html · goal.html · journal.html은 삭제하지 않고 location.replace('/asset.html?tab=…') 한 줄짜리 스텁으로 전환했다. 기존 북마크나 외부 링크가 깨지지 않도록 하면서 새 구조로 자연스럽게 리다이렉트된다.

매매일지 — 날짜별 그룹 표시

저장된 기록을 날짜 내림차순으로 그룹화해 표시한다. 같은 날짜 기록이 한 섹션에 묶이고, 섹션 헤더에 그날의 총 투자금과 예상 수익 합계가 표시된다. localStorage 키 fg-journal에 날짜 인덱스 객체로 저장.

회고

URL 쿼리 파라미터로 탭 상태를 관리하는 패턴은 이번에 처음 적용했다. 초기에는 localStorage로 마지막 탭을 기억하려 했지만, 기존 URL을 리다이렉트 스텁으로 전환하는 과정에서 "URL이 곧 상태"라는 접근이 훨씬 자연스럽다는 걸 깨달았다.

세 계산기가 독립적으로 작동하면서도 하나의 파일에 공존하는 구조는, 공유 상태가 없을 때 오히려 단순하게 유지된다는 것을 보여준다. 탭 간 데이터 연동이 필요했다면 Context 패턴을 고민했겠지만, 각 계산기는 독립적인 입력 → 출력이라 불필요했다.
