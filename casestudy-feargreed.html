공포탐욕지수

"CNN Fear & Greed Index 실시간 추적 — API부터 캔버스까지"

왜 만들었나

주식 시장의 분위기를 숫자 하나로 보고 싶었다. CNN이 제공하는 Fear & Greed Index는 변동성, 시장 모멘텀, 안전자산 수요 등 7개 지표를 종합해 0~100 사이의 수치로 시장 심리를 나타낸다. 문제는 공식 페이지가 국내에서 느리고, 데이터만 뽑아서 내 방식으로 시각화할 수 없다는 것이었다.

"내가 원하는 방식으로 보자." 게이지, 타임라인, 캘린더 히트맵, 워치리스트를 하나의 앱에 담기로 했다. 한국어/영어 전환, 다크/라이트 테마도 처음부터 설계에 포함했다.

서버 없이 Vercel 서버리스 함수 하나로 CORS 문제를 해결하고, 캔버스 API로 게이지와 타임라인을 직접 그리는 것이 핵심 기술적 도전이었다.

구현 기능 12개
탭 구성 5개

고민과 결정

설계 과정에서 맞닥뜨린 네 가지 선택지

결정 1 — CORS 프록시

고민: 브라우저에서 CNN API와 Yahoo Finance를 직접 호출하면 CORS 오류가 난다

A. Vercel 서버리스 함수 자체 프록시  |  B. corsproxy.io 같은 외부 서비스

→ 선택: A (Vercel 서버리스 프록시)

이유: 외부 프록시는 서비스 중단·정책 변경에 취약하다. 실제로 개발 중 corsproxy.io가 일시적으로 막혀 API 호출이 전부 실패한 경험이 있었다. Vercel에 api/proxy.js 하나를 두면 도메인 허용 목록도 직접 관리할 수 있고, 포트폴리오 배포 인프라와 일체화된다.

결정 2 — 차트 렌더링 전략

고민: 게이지·타임라인·인디케이터를 모두 같은 방식으로 그릴 수 있나

A. 전부 Canvas API  |  B. 게이지·타임라인은 Canvas, 인디케이터는 Chart.js

→ 선택: B (혼합 전략)

이유: 게이지(바늘+호)와 타임라인(구간 색상+오버레이)은 커스텀 렌더링이 필요해 Canvas가 맞다. 반면 인디케이터(7개 서브지표 라인차트)는 Chart.js가 제공하는 tooltip·zoom·legend가 그대로 필요하다. 도구를 목적에 맞게 쓰는 것이 최선이었다. Chart.js는 첫 탭 클릭 시 lazy-load해 초기 로딩 비용도 줄였다.

결정 3 — 다국어(i18n) 구조

고민: 한국어/영어 전환을 어떻게 관리하나

A. 인라인 TRANSLATIONS 객체 (단일 HTML)  |  B. 별도 JSON 파일 fetch

→ 선택: A (인라인 TRANSLATIONS)

이유: 단일 HTML 원칙을 지키면 파일 하나로 오프라인에서도 동작한다. JSON fetch는 네트워크 요청이 한 번 더 필요하고, 실패 시 fallback 처리가 복잡해진다. TRANSLATIONS.en / TRANSLATIONS.ko 두 객체를 나란히 두고, t('key') 함수 하나로 현재 언어 값을 꺼내는 방식으로 단순하게 처리했다.

결정 4 — 데이터 갱신 주기

고민: FGI 지수를 얼마나 자주 새로 가져올까

A. 1시간 캐시 (환율·시장 티커 포함)  |  B. 실시간 polling

→ 선택: A (1시간 캐시)

이유: CNN FGI는 분 단위로 바뀌지 않는다. 데이터 특성상 하루에 몇 번 업데이트되는 지수를 매분 fetching하면 API 부하만 늘고 사용자에게는 차이가 없다. 환율·시장 티커도 포트폴리오 참고용이므로 1시간 주기면 충분하다. 캐시는 브라우저 메모리에만 저장하고 새로고침 시 재요청한다.

데이터 흐름

fetchData()가 CNN API를 호출해 현재 FGI 점수와 히스토리 데이터를 받아온다. applyData()가 게이지 애니메이션, 히스토리 카드, 타임라인 렌더링을 순서대로 실행한다. 각 탭의 캔버스는 자신이 활성화될 때만 그린다. S&P 500 오버레이는 사용자가 토글할 때 fetchAndCacheSP500()을 한 번만 호출하고 이후 캐시를 재사용한다.

CORS 프록시 URL 인코딩

encodeURIComponent 필수 규칙

Yahoo Finance URL에 &가 포함되어 있어 인코딩 없이 프록시에 넘기면 쿼리 파라미터가 분리된다. CORS_PROXY + encodeURIComponent(url)로 전체 URL을 인코딩해야 Yahoo가 완전한 요청을 받는다. 이 규칙을 어기면 데이터는 오는데 차트가 빈다.

캔버스 렌더링 설계

게이지 — 바늘 회전 공식

게이지 바늘은 rot = angle + Math.PI / 2로 계산한다. 조건 분기 없이 일정한 오프셋을 더하는 방식이 유일하게 올바른 수식이다. 조건문을 추가하면 특정 구간에서 레이블이 뒤집히는 버그가 발생한다.

타임라인 — S&P500 nearest-neighbor 매칭

FGI 히스토리 타임스탬프(밀리초)와 Yahoo 타임스탬프(ts * 1000)를 날짜 문자열로 비교하면 타임존 오프셋으로 하루씩 밀린다. ±3일 이내 nearest-neighbor로 매칭해야 정확하다. S&P500은 자체 min/max로 0~100 정규화 후 FGI와 같은 좌표계에 그린다.

localStorage 키

언어(fg-lang), 테마(fg-theme), 워치리스트(fg-watchlist), 포트폴리오(fg-portfolio)를 각각 독립적으로 저장한다. 갤러리 허브와 테마를 공유하지 않고 앱 전용 키를 사용해 키 충돌을 방지했다.

회고

가장 어려운 부분은 CORS 프록시와 URL 인코딩이었다. Yahoo Finance URL에 &interval=1d처럼 쿼리 파라미터가 여러 개 붙을 때, 인코딩 없이 프록시에 넘기면 Yahoo가 불완전한 요청을 받아 빈 데이터를 돌려준다. 원인을 찾는 데 예상보다 훨씬 많은 시간이 걸렸다.

Canvas로 게이지를 직접 그리는 것은 좌표 수식과 씨름하는 과정이었다. 바늘 회전 공식 하나에 조건 분기를 넣었다가 특정 구간에서 레이블이 뒤집히는 버그를 만났다. 결국 Math.PI / 2 오프셋 하나로 고정하는 것이 정답이었다. "덜 영리하게 짜는 것이 더 견고하다"는 교훈이었다.
