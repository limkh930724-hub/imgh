커머스 시스템

"백화점형 이커머스 — 기획부터 구현까지"

왜 만들었나

기존 커머스 툴은 OMS·WMS·PLM 관리 도구에 가까웠다. 상품을 직접 골라 장바구니에 담고 주문하는 실제 쇼핑 경험이 없었다. "관리자가 아니라 구매자로서도 써보고 싶은 커머스"를 만들고 싶다는 생각에서 출발했다.

목표는 단순했다. 백화점처럼 여러 브랜드가 입점해 있고, 상품을 탐색하고, 장바구니에 담아 주문하고, 관리자 화면에서 재고와 매출을 확인할 수 있는 흐름을 하나의 HTML 파일 안에 담는 것.

포트폴리오 특성상 서버도, 빌드 도구도, 패키지 매니저도 없다. 단일 HTML만으로 실제 서비스처럼 동작하는 커머스를 만드는 것이 핵심 제약이자 도전이었다.

개발 기간 1일
구현 기능 10개
컴포넌트 12개
코드 ~1,200줄

고민과 결정

설계 과정에서 맞닥뜨린 네 가지 선택지

결정 1 — 기술 선택

고민: 빌드 없이 컴포넌트 기반 UI를 만들 수 있나

A. React CDN + Babel Standalone  |  B. Vanilla JS

→ 선택: A (React CDN)

이유: 상품·재고·장바구니·주문이 서로 연동되어 상태 복잡도가 높다. 컴포넌트 단위로 UI를 나누지 않으면 DOM 조작 코드가 스파게티가 된다. Babel Standalone은 빌드 도구 없이 JSX를 쓸 수 있어 포트폴리오 제약(단일 HTML)과도 양립한다.

결정 2 — 상태 관리 구조

고민: 장바구니와 재고가 여러 탭에서 동시에 읽힌다

A. 단일 StoreContext  |  B. 탭별 로컬 상태

→ 선택: A (단일 StoreContext)

이유: 재고 차감(관리자) → 품절 표시(쇼핑몰) → 주문 검증(장바구니)은 한 흐름이다. 탭마다 상태를 따로 두면 동기화 버그가 불가피하다. Context 하나로 products / inventory / cart / orders를 소유하고 각 탭은 읽기만 하는 단방향 구조를 택했다.

결정 3 — 데이터 설계

고민: 백엔드 없이 실제 DB처럼 동작하게 하려면

A. localStorage + 시드 데이터  |  B. 단순 하드코딩

→ 선택: A (localStorage + 시드 데이터)

이유: 새로고침 후에도 상태가 유지되어야 한다. 관리자에서 상품을 추가하면 쇼핑몰 탭에 즉시 반영되어야 포트폴리오 가치가 있다. INIT_PRODUCTS / INIT_INVENTORY / SEED_ORDERS로 초기값을 세팅하고, 이후 모든 변경은 localStorage에 쓰는 방식으로 실제 DB 흐름을 흉내냈다.

결정 4 — 화면 구조

고민: 기능이 많아서 페이지를 분리해야 하나

A. 단일 HTML 4탭 SPA  |  B. 기능별 멀티페이지

→ 선택: A (단일 HTML SPA)

이유: 포트폴리오의 "단일 HTML 원칙"을 유지하고 싶었다. Context를 공유하면 탭 전환만으로 홈·쇼핑몰·장바구니·관리자를 커버할 수 있다. 멀티페이지면 Context를 localStorage 경유로 연결해야 하는데, 그러면 SPA의 이점(즉각적 상태 공유)이 사라진다.

화면 설계

4탭 구조와 단방향 데이터 흐름

컴포넌트 구조

┌──────────┐ ┌──────────┐ ┌────────────┐ ┌──────────┐ │ 홈 탭 │ │ 쇼핑몰 탭 │ │ 장바구니·주문│ │ 관리자 탭 │ │ HomeTab │ │ ShopTab │ │ CartTab │ │ AdminTab │ └────┬─────┘ └────┬─────┘ └─────┬──────┘ └────┬─────┘ └─────────────┴───────────────┴───────────────┘ StoreContext products / inventory / cart / orders

데이터 흐름

StoreContext가 4개 상태(products / inventory / cart / orders)를 소유하고, 각 탭은 읽기만 한다. 쓰기는 반드시 Context의 액션 함수(addToCart, placeOrder, adjustInventory 등)를 통해서만 가능하다. 단방향 흐름 덕분에 어느 탭에서 상태가 바뀌어도 다른 탭에 즉시 반영된다.

UX 결정

모달 통합 — 단일 modal 상태 객체

상품 상세와 상품 등록·수정 폼을 별도 boolean 상태로 관리하면 두 모달이 동시에 열리는 버그가 생긴다. modal: { type: 'product' | 'productForm', product } 단일 객체 하나로 통합해 원천 차단했다. null이면 닫힘, 객체면 열림.

Toast 알림 — 우하단 3초 자동 제거

장바구니 담기, 주문 완료, 재고 부족 오류 세 가지 피드백을 모두 우하단 toast로 통일했다. alert()은 흐름을 막고, 인라인 에러 메시지는 위치를 찾아야 한다. Toast는 사용자가 하던 일을 방해하지 않으면서 결과를 즉시 알려준다. setTimeout 3초로 자동 제거.

localStorage 키 설계

기존 포트폴리오 툴과 키 충돌을 막기 위해 dept- 접두사를 통일했다. 새로고침 후 Context 초기화 시 localStorage를 읽어 마지막 상태를 복원한다. 시드 데이터(INIT_PRODUCTS 등)는 해당 키가 비어 있을 때만 적용된다.

회고

단일 HTML 안에서 React 설계 원칙(단방향 데이터 흐름, Context 분리)을 지키는 것이 가장 어려운 제약이었다. 컴포넌트가 늘어날수록 같은 파일 안에서 의존성을 관리해야 했고, 스크롤 한 번에 전체 코드를 파악할 수 있어야 했다.

빌드 도구 없이 Babel Standalone만으로 JSX를 쓰는 방식은 개발 속도와 포트폴리오 접근성 사이의 최선의 트레이드오프였다. 실제 프로젝트라면 Vite + TypeScript를 선택했겠지만, "누구나 파일 하나만 열면 바로 볼 수 있다"는 포트폴리오의 핵심 가치를 지키는 선택이었다.
